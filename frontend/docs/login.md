# 注册与登录功能设计说明

> 项目：软小筑 AI 管家  
> 版本：v0.1  
> 日期：2026-07-16  
> 技术范围：React + TypeScript 前端；FastAPI + Python 后端

## 1. 功能目标

建立可用、安全、可扩展的用户注册与登录基础能力。用户可以使用邮箱和密码创建账号、登录系统，并在登录有效期内访问 AI 管家、知识库等需要身份识别的页面。

系统不保存明文密码，前端不保存服务端密钥。

## 2. 本期范围

本期实现：

- 邮箱 + 密码注册
- 邮箱 + 密码登录
- 当前用户信息查询
- 退出登录
- 未登录页面拦截

本期暂不实现：

- 第三方登录（微信、GitHub 等）
- 短信验证码登录
- 忘记密码与邮箱找回
- 多因素认证

## 3. 用户流程

### 3.1 注册

1. 用户进入注册页。
2. 填写邮箱、密码、确认密码，并勾选同意服务条款。
3. 前端完成格式校验与两次密码一致性校验。
4. 前端调用注册接口。
5. 后端校验邮箱唯一性，生成密码哈希并创建用户。
6. 注册成功后跳转到登录页，或后续可改为直接登录。

### 3.2 登录

1. 用户进入登录页，填写邮箱和密码。
2. 前端调用登录接口。
3. 后端验证密码哈希。
4. 验证成功后签发访问令牌，并返回基本用户信息。
5. 前端保存登录状态，跳转至 AI 管家主页。

### 3.3 登录后与退出

- 页面刷新后应在令牌有效期内保持登录状态。
- 请求受保护接口时自动携带登录令牌。
- 用户主动退出后，前端清除登录状态。
- 未登录访问 AI 管家、知识库等受保护页面时，跳转至登录页；登录成功后返回原目标页面。

## 4. 页面与交互要求

### 登录页

- 邮箱输入框
- 密码输入框
- 登录按钮
- 跳转注册链接
- 错误提示使用统一文案：`邮箱或密码错误`

### 注册页

- 邮箱输入框
- 密码输入框
- 确认密码输入框
- 同意服务条款复选框
- 注册按钮
- 跳转登录链接

## 5. 校验规则

| 字段 | 前端校验 | 后端校验 |
| --- | --- | --- |
| 邮箱 | 必填，符合常用邮箱格式 | 必填、格式合法、注册时全局唯一 |
| 密码 | 必填，至少 8 位 | 至少 8 位，哈希后保存 |
| 确认密码 | 必须与密码一致 | 注册接口只接收 `password`，前端负责比对 |
| 条款同意 | 必须勾选 | 本期可记录同意时间，后续接入正式条款版本 |

## 6. 接口设计

### `POST /api/auth/register`

创建新账号。

请求体：

```json
{
  "email": "user@example.com",
  "password": "your-password"
}
```

成功响应：`201 Created`

```json
{
  "id": "user-id",
  "email": "user@example.com"
}
```

### `POST /api/auth/login`

验证账号密码并获取登录令牌。

请求体：

```json
{
  "email": "user@example.com",
  "password": "your-password"
}
```

成功响应：`200 OK`

```json
{
  "access_token": "jwt-token",
  "token_type": "bearer",
  "user": {
    "id": "user-id",
    "email": "user@example.com"
  }
}
```

### `GET /api/auth/me`

获取当前登录用户。

请求头：

```http
Authorization: Bearer <access_token>
```

成功响应：`200 OK`

```json
{
  "id": "user-id",
  "email": "user@example.com"
}
```

### `POST /api/auth/logout`

退出登录。客户端清除登录态；后续如加入刷新令牌，可由后端使刷新令牌失效。

成功响应：`204 No Content`

## 7. 数据模型

初期可使用 SQLite；部署阶段建议切换 PostgreSQL。

### `users` 用户表

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | UUID 或整数 | 用户唯一标识 |
| `email` | string，unique | 登录邮箱，统一转小写保存 |
| `password_hash` | string | 使用 Argon2 或 bcrypt 生成的密码哈希 |
| `created_at` | datetime | 注册时间 |
| `last_login_at` | datetime，可为空 | 最近一次成功登录时间 |

## 8. 安全要求

- 密码仅通过 HTTPS 传输；后端只保存 Argon2 或 bcrypt 密码哈希，禁止保存或记录明文密码。
- 登录失败统一返回“邮箱或密码错误”，避免泄露账号是否存在。
- 访问令牌使用短期 JWT；签名密钥只放在后端 `.env`，不得提交到 Git，也不得写入前端代码。
- 注册和登录接口应增加限流，降低暴力破解风险。
- 生产环境优先使用 `HttpOnly`、`Secure`、`SameSite` Cookie 保存刷新令牌。
- 本地开发阶段可先在前端内存中保存短期访问令牌，避免长期令牌暴露。

## 9. 实施顺序

1. 后端：创建 `users` 表，加入密码哈希工具，实现注册、登录和当前用户接口。
2. 前端：创建 `LoginPage`、`RegisterPage`、路由守卫和登录状态管理。
3. 联调：注册新账号并登录，通过 `/api/auth/me` 验证身份。
4. 完善：登录页错误提示、退出登录和未登录跳转。

## 10. 验收标准

- 重复邮箱不能注册。
- 合法邮箱和密码可以成功注册、登录。
- 错误密码或不存在的邮箱不能登录，且提示不泄露账号存在状态。
- 未登录用户不能访问受保护页面或接口。
- 退出登录后不能继续访问受保护页面或接口。
- 数据库和日志中均不出现明文密码。
